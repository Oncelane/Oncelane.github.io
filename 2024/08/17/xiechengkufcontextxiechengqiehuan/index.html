协程库-fcontext协程切换

发表于 2024-08-17

协程库原理之协程切换

协程分为有栈协程和无栈协程，但它们进行协程切换的核心过程是一致的。

无非就是将寄存器上的值保存到一个安全的地方，或者反过来，将之前保存好的上下文放到当前寄存器中

我们先学习ucontext的使用，再来解释如何用fcontext代替ucontext实现更高效的切换效率

使用linux下的ucontext进行协程切换非常简单，总共只需要用到三个接口和一个结构体

  1. int getcontext(ucontext_t *ucp);

    获取当前上下文

  2. void makecontext(ucontext_t *ucp, void (*func)(), int argc, …);

    初始化上下文，传入的函数指针为上下文将执行的函数体

  3. int swapcontext(ucontext_t *oucp, const ucontext_t *ucp);

    保存和切换上下文

ucontext_t结构体定义如下

typedef struct ucontext {
              struct ucontext *uc_link;	// uc_link 当前协程执行完毕后默认跳转到的下一个上下文
              sigset_t         uc_sigmask;	// 信号掩码
              stack_t          uc_stack;	// 栈指针
              ...
          } ucontext_t;

这里让gpt帮忙写一个demo帮助理解：

#include <stdio.h>
#include <stdlib.h>
#include <ucontext.h>
#include <unistd.h>

#define STACK_SIZE 8192

ucontext_t main_context;
ucontext_t ctx1, ctx2;
char stack1[STACK_SIZE];
char stack2[STACK_SIZE];

// 协程1
void coroutine1() {
    printf("Coroutine 1: Start\n");
    // 切换到协程2
    swapcontext(&ctx1, &ctx2);
    printf("Coroutine 1: End\n");
}

// 协程2
void coroutine2() {
    printf("Coroutine 2: Start\n");
    // 切换到协程1
    swapcontext(&ctx2, &ctx1);
    printf("Coroutine 2: End\n");
}

int main() {
    // 初始化协程1上下文
    getcontext(&ctx1);
    ctx1.uc_link = &main_context;  // 当协程1完成后，返回到主上下文
    ctx1.uc_stack.ss_sp = stack1;
    ctx1.uc_stack.ss_size = STACK_SIZE;
    makecontext(&ctx1, coroutine1, 0);

    // 初始化协程2上下文
    getcontext(&ctx2);
    ctx2.uc_link = &main_context;  // 当协程2完成后，返回到主上下文
    ctx2.uc_stack.ss_sp = stack2;
    ctx2.uc_stack.ss_size = STACK_SIZE;
    makecontext(&ctx2, coroutine2, 0);

    // 启动协程1
    printf("Main: Starting Coroutine 1\n");
    swapcontext(&main_context, &ctx1);

    printf("Main: All coroutines completed\n");
    return 0;
}

可以看到，用法就是使用getcontext + makecontext初始化协程的上下文结构体，

然后调用swapcontext进行协程上下文的切换和保存

上面demo的代码流为：

  1. 主线程 -> 协程1

  2. 协程1 -> 协程2

  3. 协程2 -> 协程1

  4. 协程1 -> 主协程

这就是非常经典的对称协程，也就是一个协程可以直接切换到另一个协程中，好似双方地位对等，多么美好。

对应的还有非对称协程：所有协程分为两类，唯一的主协程和许多子协程

线程中保存有唯一的主协程，因此也可以说主协程就是线程本身，主协程可以切换到任意子协程，但是子协程只能切换为主协程，也就是是说子协程1切换到子协程2的最短流程为：

  1. 协程1 -> 主协程

  2. 主协程 -> 协程2

灵活性相比demo中展示差了许多，还有额外的cpu开销

那么为什么我的协程库还是采用非对称协程设计呢？

因为对称协程在实际使用中会出现一个问题，协程切换不应该是手动的（如果是，那就是纯纯的状态机，而且是手搓的状态），也就是说协程库需要保证每一个协程都能通过某种方式自动得知它的下一个接手线程控制权的协程是谁。

如果采用类似上面对称协程的设计会导致很严重的代码复杂度，代码复杂度同样反过来影响性能。

因此下面不仅将ucontext切换方式改为fcontext，同时还将初步设计一个非对称协程的最小模型

fcontext

fcontext在1.61版本之前的api与ucontext不能说类似，只能说一摸一样，也就是说基本什么也不用做，改改函数名就能完成升级

1.61之后的boost对fcontext库做了许多改动，导致api使用方式与之前有了很大的不同，并且没有任何官方文档！！！

本人进行了为期一天的测试后基本摸清了接口用法

fcontext的接口有三个，结构体两个：

  1. transfer_t jump_fcontext( fcontext_t const to, void * vp);

  2. fcontext_t make_fcontext( void * sp, std::size_t size, void (* fn)( transfer_t) );

  3. transfer_t ontop_fcontext( fcontext_t const to, void * vp, transfer_t (* fn)( transfer_t) );

fcontext_t： 子段不重要，只需要知道它保存了协程上下文

transfer_t 的定义如下： 

struct transfer_t {
    fcontext_t  fctx; // from 协程上下文 （from！）
    void    *   data;
};

完整源码如下，明天再解析


using namespace boost::context::detail;

// 内存分配接口，可以优化成内存池
class Allocator {
public:
    static void* Alloc(size_t size) {
        return malloc(size);
    }

    static void Dealloc(void* vp, size_t size) {
        return free(vp);
    }
};

class fFiber;
static thread_local fFiber* t_threadFiber;
class fFiber {
private:
public:
    using callBack = std::function<void(void)>;

    // 无参构造，用于构造本线程的主协程
    fFiber() : m_stack(nullptr), m_stackSize(0) {
        //空的，只是需要用到m_fctx来存一些东西
    }

    // 含参构造，用于构造一个将执行cb函数的协程对象
    fFiber(callBack _cb) : m_cb(_cb), m_stackSize(1024 * 4) {
        // 使用malloc,分配协程栈，这里固定为4k大小
        // (问题1：可以猜测为什么不用new？)
        m_stack = (char*)Allocator::Alloc(m_stackSize);

        // 构造上下文，执行要执行的函数为MainFunc
        // (问题2：为什么不可以直接将MainFunc直接替换为cb？)
        m_fctx = boost::context::detail::make_fcontext(
            m_stack + m_stackSize, m_stackSize, MainFunc);
    }

    // 析构函数，释放栈内存
    ~fFiber() {
        if (m_stack) {
            Allocator::Dealloc(m_stack, m_stackSize);
        }
    }

    // 切入协程，执行cb函数
    void swapIn() {
        assert(GetThis() != this);
        auto rt = boost::context::detail::jump_fcontext(m_fctx, this);

        // 保存协程进度
        m_fctx = rt.fctx;
    }

    // 从协程切出到本线程的主协程
    void swapOut() {
        // 问题3：为什么这里不需要接受返回值，并保存写成进度？
        jump_fcontext(t_threadFiber->m_fctx, nullptr);
    }

    // MainFunc用于包装cb，执行一些初始化和善后工作
    static void MainFunc(transfer_t in) {
        // 主协程的上下文保存为 swapIn时的上下文
        t_threadFiber->m_fctx = in.fctx;

        // 拿到swainIn中传入的协程体指针
        fFiber* readyFiber = reinterpret_cast<fFiber*>(in.data);

        // 执行cb函数
        readyFiber->m_cb();

        // 默认在函数执行完毕后切回主协程
        readyFiber->swapOut();
    }

    // 初始化并获取主协程
    static fFiber* GetThis() {
        if (t_threadFiber == nullptr) {
            t_threadFiber = new fFiber();
        }
        return t_threadFiber;
    }

    fcontext_t       m_fctx;       // 协程上下文
    fFiber::callBack m_cb;         // 函数体
    char*            m_stack;      // 协程栈
    size_t           m_stackSize;  // 协程栈大小
};

void testFiberSwapOut() {
    fFiber* f1 = new fFiber([f1]() {
        printf("in f1\n");
        printf("swap out first\n");
        f1->swapOut();
        printf("in f1\n");
        printf("swap out second\n");
        f1->swapOut();
        printf("in f1\n");
        printf("exit f1\n");
    });


    printf("in main\n");
    printf("swap in first\n");
    f1->swapIn();
    printf("in main\n");
    printf("swap in second\n");
    f1->swapIn();
    printf("in main\n");
    printf("swap in third\n");
    f1->swapIn();
    printf("in main\n");
    printf("exit main\n");
}
int main() {
    testFiberSwapOut();
}
